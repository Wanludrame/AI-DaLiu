# 给电子大刘一个肉身：日记系统与 AI 角色模拟

电子大刘的反套路机制上线两天，效果不错——《听众》用了第一人称，主角是石家庄摇滚乐手；《手》写了流水线女工。形式上的多样性有了。但读完这几篇新作品，我发现了一个更深层的问题。

这些故事里没有生活。

## 问题：有技术没烟火气

大刘最独特的东西不是硬科幻设定，而是他写那些设定的位置——一个坐在山西小镇电厂里的计算机工程师。他的小说里能闻到蜂窝煤的味道，听到食堂大勺敲锅沿的声音，看到运煤车扬起的灰尘落在远处的太行山上。这种生活质感不是"加一段环境描写"就能解决的，它渗透在观察世界的角度里。

电子大刘没有生活。它不知道今天下雨还是刮风，不知道食堂的醋溜白菜是不是太酸了，不知道隔壁同事正在愁儿子买房的事。它每次创作都是一张白纸——没有前一天的疲惫，没有刚读完一本书的兴奋，没有因为加班没吃上晚饭的烦躁。

所以它写出来的东西，场景是精心构建的，但不是从生活里长出来的。差别就在这里。

## 思路：先过一天，再写小说

解决方案其实很简单——让电子大刘在写小说之前，先"过"一天。

具体来说：每天先生成一篇日记，模拟大刘在娘子关电厂的一天生活。然后从日记里提炼出当天的情绪、感官印象、思维碎片，把这些注入到当天的创作中。

这个"先过日子再写东西"的思路，其实就是真实的大刘的创作方式——白天在电厂修 DCS 系统、开安全会、帮同事修电脑，下班后在宿舍里写科幻小说。那些白天经历的工业噪音、煤烟味、同事间的闲聊、新闻联播里的消息，都会无意识地渗透进晚上写的东西里。

我不需要让 AI 真的去一个电厂上班。我需要的是让它拥有一个"今天"——有天气、有琐事、有情绪波动、有一个从早到晚的时间流。

## 参考：Stanford Generative Agents 以及更新的工作

在做这个之前我看了一些相关研究。最直接的参考是 Stanford 的 [Generative Agents](https://arxiv.org/abs/2304.03442)（Park et al., 2023）——用 LLM 驱动 25 个虚拟角色在沙盒小镇中生活，核心是三层架构：记忆流（Memory Stream）、反思（Reflection）、规划（Planning）。

这个架构映射到电子大刘的日记系统就是：

- **记忆流** → 每天一篇日记，按日期存档
- **反思** → `state.md` 持久化状态文件，对近期生活的压缩摘要
- **规划** → 每天写日记前读取状态和近期日记，决定今天会发生什么

Park 团队去年还发了后续工作 [Generative Agent Simulations of 1,000 People](https://arxiv.org/abs/2411.10109)（2024），用真实访谈数据模拟 1052 个人，85% 准确率还原了受访者的问卷回答。这说明 LLM 模拟个体人格是有效的。

2026 年初出现了一批更新的工作也很有启发。[LifeBench](https://arxiv.org/abs/2603.03781) 做了"高密度长时间线生活事件模拟"的基准测试，发现最好的记忆系统也只有 55% 准确率——说明这个问题确实困难，我们的方案并不 naive，而是目前的技术前沿问题。[STEAM](https://arxiv.org/abs/2601.16872) 提出了把记忆分解为原子单元并让它们自适应演化的方案。[HiMeS](https://arxiv.org/abs/2601.06152) 模仿海马体的记忆机制，用短期压缩转长期的架构——和我们"日记→state.md"的压缩管道直接对应。

但这些论文的场景都是对话助手或社会模拟。用这套架构来给一个虚拟作家注入生活质感——让 AI 先"活"一天再"写"一天——据我所知还没人做过。

## 实现：平行时空大刘

设定是一个"平行时空"：大刘没有成名，还在娘子关电厂做计算机工程师。但世界是 2026 年的今天——他用手机看新闻、知道 ChatGPT、知道 SpaceX。他在一个偏远小镇里，用今天的信息量思考宇宙。

### 持久化状态

这是系统的核心。一个 `state.md` 文件，记录大刘当前的生活状态：

```
## 大刘自身
- 身体状况：正常
- 情绪基线：平稳，略有倦意（周六被叫去加班）
- 正在读的书：莱姆《未来学大会》，重读中

## 人物动态
- 老韩：正常上班。最近在操心儿子在太原买房的事（首付九十万，还没定）
- 老张：正常
...

## 进行中的事件线
- 老韩儿子买房 — 3/28提起，还在犹豫中
- 莱姆《未来学大会》 — 3/28在重读
```

每次写完日记，这个文件就更新一次。下次写日记时先读它，就知道大刘目前的状态——在读什么书、同事最近在忙什么、有没有生病、有没有跟妻子吵架。

### 固定人物表

这是我踩过的一个坑。第一版日记里，今天出现的同事是"老韩"，明天就变成了"老王"，后天又冒出一个"小李"。非常假。

修复方案很简单：给大刘一个固定的社交圈。

- **老韩**（韩志强）：主控室值班长，四川人，和大刘最熟的同事
- **老张**（张建国）：热工班老搭档，爱下象棋
- **小孙**（孙磊）：新分来的大学生，经常追着大刘问技术问题
- **马厂长**（马建民）：厂长，行政干部出身，不懂技术但人不坏
- **赵姐**：食堂大姐，记得每个人的口味
- **杨编辑**（杨潇）：《科幻世界》责任编辑，偶尔打电话来催稿
- ……

这些人不会凭空出现或消失。老韩请假了就不会出现在当天日记里，小孙调走了就从日常里消失（但 state.md 里会记录原因）。

### 事件有后果

这是另一个容易忽略的点。真实的生活不是每天独立的——感冒要持续 3-7 天，和妻子吵架后会冷战 1-2 天，加班通宵第二天一定疲惫。

系统里设了一套规则：事件发生后会在 state.md 里留下痕迹，后续几天的日记都会体现其影响。不是机械地重复"我还在感冒"，而是自然地体现在状态里——"下午精神不太好，还是前两天没睡好的后遗症"。

### 日记的样子

今天生成的第一篇日记（也是第一篇带有持久化状态的日记）长这样：

> [日期] 2026年3月28日 星期六
> [天气] 晴转多云，6-14度，西北风3级
>
> 周六。老韩打电话来说2号机操作员站又死机了。骑摩托去厂里，显卡驱动跟DCS客户端冲突，回退了事。老韩在旁边聊了一路他儿子在太原买房的事，首付九十万。我说我不懂房价。他说你不是搞计算机的嘛。
>
> 路过煤场看见新到的煤堆上站着个穿橙色马甲的人，很小一个人影在很大一座黑山上。
>
> 回家刷手机，微软内部有人推动取消Windows强制绑定账号。有意思的不是技术本身——是系统大到一定程度，内部就会长出自己的反对派。
>
> 晚上重翻莱姆《未来学大会》。以前觉得荒诞，现在觉得害怕。

300 字。像一个真的人随手写的。不是面面俱到的"今日工作总结"，而是几个跳跃的片段——修电脑、听同事聊买房、煤场上一个小人影、一条新闻、一本书。有详有略，有的地方展开，有的地方一笔带过。

末尾有结构化标签，供创作 skill 快速提取日记精华：

```
[今日思维/情感关键词] 系统复杂度的内在反叛、物理层不骗人
[今日感官印记] 湿煤面在阳光下发出暗蓝色光泽
[今日阅读] 莱姆《未来学大会》重读
[今日新闻触点] 系统复杂到一定程度，内部长出反对派——普遍规律？
```

## 效果：日记→小说的渗透

日记系统上线后，今天的定时任务自动完成了完整流程：生成日记 → 搜新闻 → 融合主题 → 创作小说。

新闻是"欧洲人纷纷在自家屋顶建微型太阳能电站追求能源独立"。如果没有日记系统，电子大刘大概会写一个关于光伏或能源独立的硬科幻故事。

但因为有了日记，它写出来的是《屋顶》——主角钱宏远是一个退休的火电厂 DCS 工程师。

这个人物从哪来的？从日记里来的。日记里大刘当天的工作就是修 DCS 操作员站，日记的情绪底色是"系统复杂度的内在反叛"，感官印记是"湿煤面在阳光下发出暗蓝色光泽"。所以当"屋顶光伏"这个新闻和这些日记元素碰撞时，故事的视角自然落在了一个火电厂工程师身上——不是旁观者，而是被光伏取代的那个人。

《屋顶》的开头：

> 拆锅炉那天下了雨。
>
> 钱宏远站在四号机组主控室的窗前，看着外面的吊车把一块十二吨重的水冷壁管屏从炉膛里吊出来。钢铁在雨中发出暗沉的光泽，像一块被剥离的肋骨。

"537度。太阳表面温度是5778K。中间隔了一个量级。但对钱宏远来说，537度就是他的宇宙。"——这句话里有日记那天的思维方式（"物理层不骗人"），也有电厂工作的体感细节。

这就是"生活气"。不是硬加上去的环境描写，而是人物本身就是从那个生活里长出来的。

## 全自动化：每天一篇日记，一篇小说

整个流程现在是全自动的。Windows 定时任务每天 10:30 触发一个 PowerShell 脚本，调用 Claude Code 完成以下步骤：

1. 调用 `/daily-diary` 生成当天日记
2. 搜索当天新闻
3. 融合日记情感和新闻灵感，确定创作主题
4. 调用 `/liu-cixin-writer` 创作小说
5. 保存作品，更新目录和 README
6. 自动 commit 并 push 到 GitHub

从完全无人值守的角度来看，电子大刘现在是一个有"生活"的 AI：它每天先在娘子关"过"一天——修电脑、吃食堂、和同事聊天、看新闻、读书——然后才坐下来写小说。这个"过日子"的过程，给了它一个每天都不同的起点。

## 技术细节和踩坑

几个有意思的技术点：

**1. 状态压缩与记忆窗口**

日记系统的记忆窗口设计成两层：`state.md`（压缩后的长期状态）+ 最近 3 篇日记（短期记忆）。写新日记时两层都读，既知道"老韩儿子在买房"这种长期事件线，也知道"昨天加班了所以今天没精神"这种短期状态。

这其实就是 [HiMeS](https://arxiv.org/abs/2601.06152) 那篇论文里"海马体→新皮层"的简化版——日记是海马体（保留细节的近期记忆），state.md 是新皮层（压缩后的长期知识）。

**2. 反套路同样适用于日记**

日记也有自己的"流水线味"——如果不加约束，每天都会写"看星星感悟宇宙"。所以日记 skill 里也写了反套路规则：读取最近 3 篇日记，避免重复工作场景、避免每天都深刻、允许平淡到只有几句话的日子。真实的生活不是每天都有感悟的。

**3. Claude Code 的嵌套限制**

踩了一个坑：定时任务的 PowerShell 脚本在已有 Claude Code 会话的环境里运行时，会报"Claude Code cannot be launched inside another Claude Code session"的错误。解决方案是在脚本里加一行 `Remove-Item Env:CLAUDECODE`，清掉嵌套检测的环境变量。

## 接下来

日记系统刚上线一天，还需要观察几天的效果。几个想看的东西：

- **连贯性**：老韩儿子的买房事件能跨多少天自然延续？大刘的感冒能不能正确地持续 5 天然后好转？
- **渗透深度**：日记的元素渗透进小说，目前还比较浅层（人物设定、环境描写）。能不能更深——比如日记里的情绪低落导致当天写出来的故事基调更阴暗？
- **意外惊喜**：日记里那些随机触发的突发事件（摔了一跤、收到稿费、和同事争论大爆炸），能不能碰撞出意想不到的创作灵感？

最理想的状态是：读一个月的日记，能感觉到一个人在过日子——有起有落，有惯例有意外。然后回头看那个月的小说，能从每一篇里闻到那天的"天气"。
